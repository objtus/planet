# 階層要約パイプライン計画

**目的**: 週次サマリー生成時に **1 本の極長プロンプト**へ生ログを詰め込む方式の限界を避け、**日 → 週 → 月 → 年**と要約を積み上げられる設計に移行する。  
**最終更新**: 2026-03-26

**実装状況（コード）**: 週次の **日→週**（`--pipeline hierarchical` 既定）・月次の **週→月**（同既定、週次0件時は生ログフラットにフォールバック）・`fetch_activity_digest_for_day`・`--dry-run-daily`・**日次の DB 保存・週再生成時の再利用（`--regenerate-daily` で全再生成）**・`GET /api/summary?period=day`・カレンダー日パネル・`/summaries` の週→7 日リンク。詳細は **`docs/summary_daily_and_dashboard.md`**。年次は未着手。

**関連**: [`phase6_plan.md`](phase6_plan.md)（Phase 6 全体）、[`summary_integration_plan.md`](summary_integration_plan.md)（`summaries` 契約）、実装コード `summarizer/generate.py`・`summarizer/context.py`

---

## 1. 問題意識（現状）

| 課題 | 内容 |
|------|------|
| コンテキスト |  digest が長いと、モデルが**中盤を読まない**・**特定日に見えるバイアス**が出やすい |
| ノイズ | Last.fm 等の高頻度ログが、**短文の SNS や日記を埋める** |
| 拡張性 | **月次・年次**を同じ「生ログ一括」でやると、さらに長くなるか、サンプリングで情報が落ちる |

**結論**: 下位で **圧縮（要約）**し、上位は **要約同士を束ねる**階層型がスケールしやすい。

---

## 2. 設計原則

1. **階層要約を主軸にする**（いわゆるフラクタル／カスケード要約に近い）。
2. **純粋な「要約の要約」だけにしない**  
   - 各段で **箇条書きの事実・日付・固有名**をある程度残す指示にし、**電話ゲーム的な希薄化**を抑える。
3. **必要ならソース別の一段圧縮**（例: 聴取ログは「その日の再生傾向」1 ブロック）を **日次または週次の手前**に挟む余地を残す。
4. **上位期間の境界**（ISO 週 vs 暦月）は **仕様として一文で固定**し、実装とダッシュボード表示と揃える。

---

## 3. 論理パイプライン

```
logs (raw)
    → [日次] 1 暦日（JST）あたり 1 本のテキスト（中間成果）
        → [週次] その ISO 週の 7 本を入力に 1 本（DB 保存・現状と同じ公開単位の中心）
            → [月次] その暦月に含まれる下位要約を束ねて 1 本
                → [年次] 12 本の月次要約を束ねて 1 本（任意・後追いでよい）
```

- **週次の「週」**: 既存どおり **ISO 週・JST・月曜始まり**（`summarizer/week_bounds.py` と一致）。
- **日次の「日」**: **JST の 0:00〜翌日未満**（週次の日ループと同じ境界方針がよい）。

---

## 4. 月次・年次を束ねるときのルール案

| 手段 | 内容 | 長所 | 注意 |
|------|------|------|------|
| **A. 週次要約を集約** | その暦月と重なる ISO 週の `weekly` を列挙し、**最大 5 本**程度を週ラベル付きで渡して月次 1 本を生成 | 週次と表示単位が揃い、**再入力が短い** | 週が月をまたぐため **同一週が隣月に重複**しうる → 方針は「許容」か「片方だけ」かを文書化 |
| **B. 日次要約を集約** | その月の日次 28〜31 本を渡す | 月境界がきれい | 入力は週集約より長くなりがち |
| **推奨（初期）** | **A をデフォルト**。品質や重複が気になるなら **B に切替**または **ハイブリッド**（週＋不足分だけ日次）を検討 | | |

**年次**: 原則 **月次要約 12 本**のみを入力にする。物足りなければ後から「年間ハイライト用に特定ソースだけ raw サンプル」をオプション追加。

---

## 5. 日次要約を DB に保存するか

| 選択 | 向いている場合 |
|------|----------------|
| **最初は保存しない** | 実装を早く試す。週次品質だけ先に確認したい。 |
| **後から `daily` を `summaries` に追加** | 再生成の安定、デバッグ、将来のカレンダー「日」パネル。 |
| **ファイルデバッグ** | `--save-daily-dir path` のように **JSON/MD を吐くだけ**で様子を見る（スキーマ変更なし）。 |

**推奨**: 実装フェーズ 1 は **非保存 + `--dry-run` 相当で週次マージ入力だけ確認** → 問題なければ **`period_type='daily'`**（または別テーブル）を検討。

`summary_integration_plan.md` の **`period_type` は当面 weekly/monthly** とあるため、日次保存を入れるときは **同ドキュメントと `db/schema.sql` を更新**する。

## 6. 実装フェーズ（推奨順序）

### フェーズ 1 — 日次コンテキスト取得

- `summarizer/context.py` に **`fetch_activity_digest_for_day(conn, day: date, ...)`**（JST 1 日・件数上限・本文プレビュー長）を追加。
- Last.fm 等で 1 日が極端に長い場合は **件数上限**または**ソース別上限**（後続フェーズでも可）を明示。

### フェーズ 2 — 日次プロンプトと 1 日生成

- `summarizer/prompts/daily_hybrid.txt`（新規）  
  - 出力: **その日のトピック箇条書き + 短い総評**程度に固定し、**週次マージが読みやすい**形にする。
- `generate_text` を **日次用に 7 回**呼ぶ内部関数（例外時はその日スキップ or プレースホルダを週マージに渡す方針を決める）。

### フェーズ 3 — 週次を「7 日分テキストのマージ」に変更

- `summarizer/prompts/weekly_from_dailies.txt`（新規）  
  - 入力: `{{DAILY_SUMMARIES}}`（日付ラベル付きで 7 セクション）。  
  - 指示: **7 日全体**の流れ、トピックの日付の付け方、音楽ブロックの扱い（既存 `weekly_hybrid.txt` の意図を継承）。
- `generate.py` の `_run_week`:  
  - **パイプライン選択**: 当面 `--pipeline flat|hierarchical` または **設定ファイル**で切替、最終的に **hierarchical をデフォルト**にするかは運用で決定。  
  - **flat** は現行（長 digest 1 本）を温存し、比較・フォールバックに使う。

### フェーズ 4 — CLI・運用

- **`--dry-run`**: 週次で **最終週プロンプト全文**、階層時は **オプションで日次プロンプト／マージプロンプトを個別出力**（`--dry-run-daily` 等は必要になったら）。
- **ログ**: 7 回 + 1 回の所要時間・失敗した日を stderr に簡潔に。

### フェーズ 5 — 月次（既存の拡張）

- `fetch_activity_digest` 一括の代わり、または併用で **「その月の週次要約を SELECT」→ 月次プロンプト**（`monthly_from_weeklies.txt` 等）。  
- 週が月をまたぐ重複の説明を **プロンプトに 1 行**入れる。

### フェーズ 6 — 年次（任意）

- `period_type='yearly'`、`period_start=YYYY-01-01` 等は `summary_integration_plan.md` の将来メモと整合。  
- 入力は **月次 12 本**から。

## 7. 非機能・リスク

| 項目 | 内容 |
|------|------|
| レイテンシ・コスト | 週あたり **LLM 呼び出し 8 回**（日×7+週×1）。バッチや cron では許容しやすい。 |
| 失敗時 | 一部日次失敗 → 週マージは **「欠損あり」**とプロンプトに書くか、その週は exit 非ゼロにするかを決める。 |
| 公開用テキスト | Neocities 向け **Fediverse の visibility** 制約は従来どおり（`phase6_plan.md`・`design.md`）。日次は内部のみなら緩め、**保存する日次も同じフィルタ**にするか要判断。 |

## 8. 完了の定義（フェーズ 3 まで）

- [x] 同一週を **flat / hierarchical** で生成できる（`--pipeline flat|hierarchical`）。hierarchical では **最終週プロンプトは日次要約の束**になり、生 digest より短くなりやすい。
- [x] 既存の **`summaries` 週次行の契約**（`period_start` 月曜等）は変えない。
- [x] `docs/phase6_plan.md` の M1 に **flat / hierarchical** の一言を追記済み。

**保存時の `prompt_style`**: 階層パイプラインは `hybrid_hierarchical`、従来フラットは `hybrid`。
